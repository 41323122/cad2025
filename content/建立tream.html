建立tream

建立tream以下為製做過程

1. 環境初始化 (僅需做一次)

安裝環境。

           安裝套件： ./init_replit

           預覽網頁： python main.py

組員在自己的分支 (Branch) 上工作，避免直接弄亂主分支 (main)。

  • 動作 指令 說明
    同步進度

    git checkout main


    git pull origin main

    		 開工前，先回主分支抓取大家更新後的成果。
    開啟新任務 git checkout -b <分支名> (僅第一次) 建立並切換到你的功能分支。
    切換分支 git checkout <分支名> 後續編輯時，確保自己在正確的分支上。
    檢查狀態 git status 確認哪些檔案被修改了。
    提交紀錄

    git add .


    git commit -m "描述你的修改"

    		 將修改存入本地倉庫。
    推送成果 git push -u origin <分支名> 將你的分支推送到 GitHub，讓組長能看到。

2. 組長：審核與合併流程

組長負責將大家的程式碼整合到 main 分支。

動作 指令 說明
更新資訊 git fetch origin 抓取雲端所有分支的最新狀態。
檢查組員進度

git branch -a


git log --oneline

 查看目前有哪些分支以及提交紀錄。
比較差異 git diff main origin/<組員分支> 在合併前，先看組員寫了什麼。
本地測試

git checkout <組員分支>


python main2.py

 切換到組員的分支並執行，確保網頁跑得起來。
正式合併

git checkout main


git merge origin/<組員分支>

 切換回主分支，並把組員的成果合併進來。
同步雲端 git push origin main 將合併後的最終版本推送到雲端。

再來是進到網站編輯前記得git pull origin main拉取最新的資料不然後面合併會看不到東西

心得

當初建立 Team 專案時，我以為只要大家建立好帳號、開好分支，剩下的就只是把程式碼丟上去就好。沒想到真正開始合併時，才發現後面有一堆事情要處理。

我們遇到最大的問題是多人 Git Pull 時，組員間的資料無法同步。後來才發現，原因出在我們太心急，沒有先拉取（Pull）最新的進度就直接進行編輯。這導致大家在不同的版本上工作，合併時出現了許多錯誤和衝突。這次經驗讓我深刻體會到，Git 協作不只是寫程式，更是一種邏輯的同步。雖然遇到很多錯誤，但現在我終於知道「先拉再寫」才是團隊開發最重要的規矩。
